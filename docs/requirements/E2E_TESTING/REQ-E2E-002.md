# REQ-E2E-002: Zitadel Local Instance Auto-Configuration

## Status: MISSING
## Priority: HIGH
## Phase: 10

## Description

The local E2E stack shall automatically provision a Zitadel instance with the OIDC application, roles, and test users required for RTMX development. The provisioning shall be idempotent and reproducible, using the Zitadel Management API to configure the instance on first boot.

## Rationale

Developers should not need to manually configure an identity provider to work on auth features. Automated provisioning ensures every developer gets an identical setup and eliminates "works on my machine" auth issues. The same provisioning logic can be adapted for Zarf package initialization.

## Acceptance Criteria

- [ ] Init script provisions Zitadel via Management API on first boot
- [ ] Provisioned resources:
  - Project: `RTMX`
  - Application: `rtmx-cli` (Native type, PKCE, no client secret)
  - Redirect URIs: `http://localhost:*/callback`
  - Roles: `rtmx:admin`, `rtmx:user`, `rtmx:readonly`
  - Test users: `alice@test.rtmx.ai` (admin), `bob@test.rtmx.ai` (user)
- [ ] Provisioning is idempotent (safe to re-run)
- [ ] Client ID and test credentials written to `.env.e2e` for test consumption
- [ ] Provisioning completes within 30 seconds after Zitadel is healthy
- [ ] Script validates Zitadel health before attempting provisioning
- [ ] Provisioning errors are clearly reported with remediation hints
- [ ] Test tokens can be obtained programmatically for headless E2E tests (Resource Owner Password Grant or machine-to-machine client credentials for test automation)

## Technical Notes

### Zitadel Management API

```python
# e2e/init/zitadel-seed.py
import requests

ZITADEL_URL = "http://localhost:8080"

# 1. Authenticate as machine user (PAT or service account)
# 2. Create project
# 3. Create OIDC application
# 4. Create roles
# 5. Create test users
# 6. Assign roles to users
# 7. Write .env.e2e with client_id and test credentials
```

### Headless Test Auth

For automated E2E tests that can't open a browser:
- Create a machine user with client credentials grant
- Or use Resource Owner Password Grant for test users (enabled only in local instance)
- Tests use `conftest.py` fixtures that obtain tokens automatically

### Configuration Output

```bash
# .env.e2e (generated by provisioning)
RTMX_AUTH_ISSUER=http://localhost:8080
RTMX_AUTH_CLIENT_ID=284759382...
RTMX_TEST_USER_ALICE=alice@test.rtmx.ai
RTMX_TEST_USER_ALICE_PASSWORD=...
RTMX_TEST_USER_BOB=bob@test.rtmx.ai
RTMX_TEST_USER_BOB_PASSWORD=...
RTMX_TEST_MACHINE_CLIENT_ID=...
RTMX_TEST_MACHINE_CLIENT_SECRET=...
```
